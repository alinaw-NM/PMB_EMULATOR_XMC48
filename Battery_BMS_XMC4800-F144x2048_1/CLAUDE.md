# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build System

This project is built exclusively through **Infineon DAVE IDE 4.5** (Eclipse CDT Managed Build). There is no standalone Makefile; the build is driven by `.cproject` configuration.

- **Build:** Project → Build Project in DAVE IDE (invokes `arm-none-eabi-gcc`)
- **Flash/Debug:** Run → Debug using the built-in J-Link debugger configuration
- **Build output:** `Debug/` directory (ELF, HEX, MAP files)
- **Debug vs Release:** Two configurations exist — Debug adds `-DXMC_DEBUG_ENABLE` and disables optimizations; Release uses `-Os`

**Do not edit files in `Dave/Generated/`** — these are regenerated by the DAVE APP configurator and any manual changes will be overwritten.

## Architecture

### Hardware Target
- **MCU:** Infineon XMC4800-F144x2048 (ARM Cortex-M4 @ 144 MHz, 2 MB Flash, 144-pin)
- **Board:** XMC4800 Relax EtherCAT Kit
- **RS485 transceiver:** THVD1406DR (auto-direction, receiver permanently enabled)
- **UART pins:** TX=P2.14, RX=P2.15 (USIC1_CH0 = `UART_0` DAVE APP); RS485 DE = P5.8 (`RS485_DE` DIGITAL_IO APP)

### DAVE Framework
`DAVE_Init()` (called first in `main()`) initializes all configured APPs: `UART_0`, `RS485_DE`, `CLOCK_XMC4`, `CPU_CTRL_XMC4`, `SYSTIMER`, `INTERRUPT`. APP handles (e.g. `UART_0`) are declared in `Dave/Generated/` and expose the underlying `XMC_USIC_CH_t *channel` used by XMCLib calls.

### RS485 / UART I/O Model
The code uses **polling mode** — USIC1 FIFO interrupts are explicitly disabled in `main()` (`NVIC_DisableIRQ(90/91)`). Because the THVD1406DR receiver is always on, every transmitted byte echoes back into the RXFIFO. The pattern for every RS485 transaction is:

1. Flush RXFIFO before TX
2. Write bytes to TXFIFO, wait for TXFIFO empty + `TRANSMISSION_IDLE`
3. `delay_ms(10)` — allow auto-direction turnaround to complete
4. Flush RXFIFO again — discards the TX echo
5. Poll RXFIFO for the BMS response with a countdown timeout

### Source File Roles
| File | Purpose |
|------|---------|
| `main.c` | Diagnostic loop: sends one analog-read request every 2 s, collects and hex-dumps the BMS response via `XMC_DEBUG` (SWO) |
| `scud_bms.c` | Full SCUD BMS protocol driver — frame encode/decode, checksum, `SCUD_ReadAnalog()`, `SCUD_ReadAlarms()`, `SCUD_PrintAnalog()` |
| `scud_bms.h` | Public API: `ScudAnalog_t`, `ScudAlarms_t`, `ScudStatus_t` |
| `scud_bms.py` | Python equivalent of the protocol — useful for PC-side testing/validation against the same BMS |

### SCUD BMS Protocol
ASCII-hex framing (YDT 1363.3-style, ref: `SCUD BMS485 协议 A2 修订版 20251015.pdf`):
```
~ VER(2) ADR(2) CID1(2) CID2(2) LENGTH(4) INFO(LENID bytes) CHKSUM(4) \r
```
- `VER=0x22`, `ADR=0x00`, `CID1=0x4A`
- `CID2=0x42` → analog telemetry (`ScudAnalog_t`); `CID2=0x44` → alarm flags (`ScudAlarms_t`)
- `LENGTH` = `(LCHKSUM << 12) | LENID` (4-bit length checksum + 12-bit data length)
- `CHKSUM` = 16-bit two's-complement of all ASCII bytes between `~` and `CHKSUM`
- Timeout constant `SCUD_TIMEOUT_TICKS = 36000000` ≈ 1 s at 144 MHz (~4 cycles/iteration)

### Debugging
`XMC_DEBUG(fmt, ...)` writes to SWO (ITM trace); requires `initialise_monitor_handles()` at startup and a debugger with SWO configured at 4 MHz. Output is visible in the DAVE IDE SWO viewer or J-Link SWO viewer. The macro compiles to nothing in Release builds (no `XMC_DEBUG_ENABLE` defined).
