# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

Embedded firmware project for the **Infineon XMC4800 IoT Connectivity Kit** (target `XMC4800-F100x2048`, ARM Cortex-M4 with FPU). It is an Eclipse CDT / DAVE IDE 4.5.0 project that exercises the MultiCAN peripheral via DAVE-generated APPs. There are sibling projects in the parent workspace (`Blinking_led`, `MultiCAN`) but this CLAUDE.md only covers `Emulator_CAN`.

## Build & flash

There is no command-line build script — the canonical build is driven by Eclipse-managed makefiles defined in `.cproject`:

- **Toolchain:** ARM-GCC (`com.ifx.xmc4000.appRelease.toolChain`), float ABI `softfp`, FPU `fpv4-sp-d16`, `-D XMC4800_F100x2048`, `--specs=nosys.specs`.
- **Configurations:** `Release` (optimization `-Os`) and `Debug`. Build artifacts land in `Release/` or `Debug/` next to this file.
- **Linker script:** `linker_script.ld` (top-level). Startup is `Startup/startup_XMC4800.S` + `Startup/system_XMC4800.c`.
- **To build / clean / flash:** use the DAVE IDE GUI (Project → Build, Run → Debug). There is no Makefile checked in — Eclipse regenerates `makefile` under the active configuration directory.
- **No test suite, no linter config.** Firmware is validated by flashing to hardware.

If a headless build is ever required, invoke `eclipsec.exe -nosplash -application org.eclipse.cdt.managedbuilder.core.headlessbuild -import . -build Emulator_CAN/Release` from the DAVE IDE install directory — but the project is normally built interactively.

## Architecture

The codebase is a thin user-application layer on top of DAVE-generated peripheral configuration. Understanding what is generated vs. hand-written is the most important thing.

### Layers

```
main.c                       ← user code (hand-written; calls DAVE_Init then idle loop)
  └─ Dave/Generated/         ← REGENERATED BY DAVE — do not hand-edit
       ├─ DAVE.c / DAVE.h    ← top-level init dispatcher; sequences APP Init calls
       ├─ CLOCK_XMC4/        ← clock tree configuration APP
       ├─ GLOBAL_CAN/        ← MultiCAN module-level clock/frequency setup
       └─ MULTICAN_CONFIG/   ← per-node + message-object configuration
  └─ Libraries/
       ├─ XMCLib/            ← Infineon XMC Peripheral Library (xmc_can.c, xmc_gpio.c, …)
       ├─ CMSIS/             ← ARM CMSIS + Infineon XMC4800 device headers
       └─ Newlib/syscalls.c  ← minimal libc stubs
  └─ Startup/                ← vector table + SystemInit
  └─ linker_script.ld
```

### DAVE APP model (critical to understand before editing)

- Each subfolder under `Dave/Generated/` is one DAVE **APP** instance. Generated files come in triples: `<app>.c/.h` (API), `<app>_conf.c/.h` (configuration constants), `<app>_extern.h` (instance handles like `MULTICAN_CONFIG_0`, `GLOBAL_CAN_0`).
- `DAVE_Init()` in `Dave/Generated/DAVE.c` is the single entry point and is declared `__WEAK` — the user may override it in `main.c` to change init ordering, but currently it just calls `CLOCK_XMC4_Init` then `MULTICAN_CONFIG_Init` (which internally calls `GLOBAL_CAN_Init`).
- The MultiCAN configuration (baudrate, nodes, message objects, FIFOs, gateway, RX/TX service requests) lives in `Dave/Generated/MULTICAN_CONFIG/multican_config_conf.c`. `NUM_MSGOBJECTS` and `NODE_n_ENABLED` macros in `multican_config_conf.h` drive the size of `MULTICAN_CONFIG_t::lmobj_ptr[]` and which `nodeN` fields are emitted.
- The real configuration source of truth is `Dave/Model/` (DAVE's project model) and `Dave/Generated/Config.xml`. **Edits to `Dave/Generated/*.c/.h` will be overwritten** the next time the user clicks "Generate Code" in DAVE. If a configuration value needs to change (e.g. CAN baud rate, message object IDs, pin assignments), the user must do that in DAVE's GUI and regenerate — Claude should not patch generated files directly except as a temporary diagnostic.

### Where to put user code

- `main.c` — application entry, after `DAVE_Init()` returns success. The placeholder `while(1U)` is where periodic logic, ISR-driven state machines, and CAN TX/RX calls (using `XMC_CAN_MO_Transmit`, the `MULTICAN_CONFIG_NODE_LMO_t` structures, etc.) belong.
- Interrupt handlers: ISR symbols (e.g. `CAN0_0_IRQHandler`) are weakly defined in `Startup/startup_XMC4800.S`. To handle a CAN service request, define a strong override in user code; the service-request number is configured per node/MO in DAVE.
- New non-generated source files can sit at the project root or in a new top-level folder — `.cproject` sources from `/` and excludes only `Dave/Model`, so anything added at the top level is picked up automatically.

### Include paths (already configured in `.cproject`)

`${ProjName}/`, `${ProjName}/Dave/Generated`, `${ProjName}/Libraries`, `${ProjName}/Libraries/XMCLib/inc`, `${ProjName}/Libraries/CMSIS/Include`, `${ProjName}/Libraries/CMSIS/Infineon/XMC4800_series/Include`. New user headers under the project root are includable as `"foo.h"`.

## Conventions specific to this project

- **Never modify files under `Dave/Generated/`** — they carry the "This file is generated by DAVE. Any manual modification done to this file will be lost when the code is regenerated." banner. Configuration changes go through the DAVE GUI; code that needs to live alongside generated APPs belongs in `main.c` or new user files.
- `Dave/Model/` should be treated similarly — it's the serialized DAVE project state, not human-edited.
- The `solver.log` at the project root is DAVE's pin/resource solver output; it can be regenerated and is not load-bearing.
